# 知构软考管理员角色鉴权设计

日期：2026-07-15

状态：已确认，待实施

范围：用户角色、认证响应、管理端后端鉴权、管理用途 OSS 上传鉴权、前端管理入口与路由保护、现有数据库迁移

## 1. 背景与目标

当前项目没有用户角色字段。`AdminController` 的所有接口只验证用户是否已登录，因此任何活动用户都可以读取用户列表、修改或删除用户，以及新增、修改或删除题目。前端同样会向所有登录用户展示后台入口，直接访问 `/admin/**` 也没有角色保护。

本次改造的目标是建立最小但真实的管理员权限边界：

- 用户只有 `USER` 和 `ADMIN` 两种角色。
- 新注册用户始终为 `USER`，客户端不能指定角色。
- 只有数据库中当前角色为 `ADMIN` 的活动用户可以调用 `/admin/**`。
- 只有数据库中当前角色为 `ADMIN` 的活动用户可以调用仅供后台页面使用的 `POST /oss/upload`。
- 管理员角色不通过后台界面授予；首版只通过受控数据库迁移授予。
- 当前已创建的 `admin@aisoftoj.local` 被迁移为 `ADMIN`，其余现有用户保持 `USER`。
- 前端隐藏普通用户的后台入口，并阻止其直接访问管理端路由。
- 后端始终是权限判定的唯一安全边界，前端状态不可提升权限。

## 2. 方案选择

采用 `user.role` 单角色字符串字段，不采用布尔 `is_admin`，也不引入完整 RBAC 表。

- 相比 `is_admin`，`role` 能在未来增加运营或题库编辑角色时保持清晰语义。
- 相比角色、权限、关联表组成的 RBAC，单角色字段更符合当前只有普通用户和管理员的实际需求。
- 角色只服务于管理端访问控制，不改变现有练习、错题、论文和个人中心的数据归属规则。

## 3. 数据模型与迁移

### 3.1 新字段

`user` 表新增：

```sql
`role` varchar(16) COLLATE utf8mb4_bin NOT NULL DEFAULT 'USER' COMMENT '用户角色：USER-普通用户，ADMIN-管理员'
```

合法值只有：

- `USER`
- `ADMIN`

代码对空值、未知值和大小写不一致的值一律按非管理员处理。管理员判断只接受精确的 `ADMIN`。

### 3.2 新数据库

根目录 `db_schema.sql` 的 `user` 建表语句直接包含 `role` 字段，默认值为 `USER`。脚本不插入默认管理员或明文密码。

### 3.3 现有数据库

新增单次执行的版本化 SQL 文件：

`db_migrations/20260715_add_user_role.sql`

迁移顺序为：

1. 备份当前数据库。
2. 给 `user` 表添加非空且默认值为 `USER` 的 `role` 字段，已有记录自动成为 `USER`。
3. 将邮箱精确等于 `admin@aisoftoj.local`、未删除的账号更新为 `ADMIN`。
4. 查询并人工或脚本验证恰好存在一个目标管理员，且其他账号均为 `USER`。
5. 数据库迁移成功后再部署依赖该字段的新后端。

如果目标管理员不存在或匹配数量不是 1，部署验证失败，不应继续开放管理端。角色迁移不修改密码、启用状态、删除状态或用户业务数据。

全新数据库不自动插入管理员或预置密码。首次部署时，在安全组仅允许维护者访问、网站尚未公开的条件下，通过现有注册接口创建 `admin@aisoftoj.local`，随后执行受控 SQL 将该账号更新为 `ADMIN`，并验证恰好一条记录受到影响。正式开放前必须修改初始密码。公开注册接口和应用启动逻辑均不得自动授予首个注册用户管理员权限。

迁移回滚时优先回滚应用版本，保留新增字段；旧版本不会读取该字段。不要在紧急回滚时删除字段或恢复整库，以免扩大数据风险。

## 4. 后端设计

### 4.1 用户实体与角色常量

`User` 增加字符串属性 `role`。后端定义集中式角色常量或枚举，业务代码不散落角色字符串。

注册流程显式设置 `USER`，即使未来请求 DTO 被扩展，也不得信任客户端提交的角色。微信或其他新增注册方式也必须使用同一默认规则。

### 4.2 认证响应

`AuthUserDTO` 增加非空 `role` 字段。以下响应都返回数据库中的当前角色：

- `POST /auth/login`
- `POST /auth/register`
- `GET /auth/me`

输出规则为：数据库值精确等于 `ADMIN` 时返回 `ADMIN`，其他情况统一返回 `USER`。这样可以保证旧数据或异常数据遵循最小权限原则。

JWT 继续只包含 `userId`、签发时间和过期时间，不加入角色。每次管理端请求均实时查询活动用户和角色，因此禁用、删除或降权立即生效，无需等待旧 JWT 过期。

### 4.3 管理员校验

`AuthService` 新增：

```java
Integer requireAdmin(String authorization);
```

校验流程：

1. 复用现有 Bearer Token 规范化、JWT 签名、必要载荷和过期时间校验。
2. 根据 `userId` 查询用户。
3. 用户不存在、已删除或已禁用时抛出 `UnauthorizedException`，返回 HTTP 401。
4. 用户有效但角色不是精确的 `ADMIN` 时抛出 `ForbiddenException("需要管理员权限")`，返回 HTTP 403。
5. 校验成功时返回管理员用户 ID，便于后续审计或管理操作使用。

401 表示身份无效，403 表示身份有效但权限不足。继续复用现有 `GlobalExceptionHandler` 的真实 HTTP 状态，不把权限错误包装成 HTTP 200。

### 4.4 管理接口

`AdminController` 的每一个 `/admin/**` 端点都必须在调用 `AdminService` 之前执行 `requireAdmin()`。现有私有 `requireLogin()` 替换为 `requireAdmin()`，不能遗漏只读或筛选辅助接口。

`AdminOssUpload` 是 `POST /oss/upload` 的唯一前端调用方，因此该接口属于管理用途接口。`OssController.upload()` 必须把现有登录校验替换为 `requireAdmin()`，并在读取文件或调用 `OssService` 前完成校验。路由路径本轮保持 `/oss/upload`，避免无关的 API 迁移；其安全规则与 `/admin/**` 相同：无效身份返回 401，活动普通用户返回 403。未来如新增普通用户上传功能，必须使用独立端点和权限、类型、目录及配额规则，不能复用此管理上传接口。

本轮不把角色修改能力加入 `AdminUserUpdateRequest`，也不新增角色管理 API。管理员不能通过现有后台把普通用户提升为管理员。

### 4.5 安全不变量

- 客户端注册、登录请求和本地存储均不能决定角色。
- 前端即使被篡改为显示后台页面，所有管理 API 仍返回 403。
- 普通用户的合法 JWT 不能调用任何 `/admin/**`。
- 普通用户的合法 JWT 不能调用管理用途的 `POST /oss/upload`。
- 管理员权限以请求时数据库状态为准，不以 JWT 或缓存状态为准。
- 空角色、未知角色、大小写错误角色全部拒绝管理权限。

## 5. 前端设计

### 5.1 用户类型与认证状态

`User` 类型增加：

```ts
role: 'USER' | 'ADMIN';
```

登录、注册和 `/auth/me` 的响应继续由现有 `useAuth` 保存。已有浏览器 `localStorage` 中可能存在不含 `role` 的旧用户对象；在认证初始化完成前不能据此开放后台。`checkAuthStatus()` 从 `/auth/me` 获取最新角色后才允许后台守卫作出放行判断。

如果网络异常导致无法刷新认证状态，后台保持关闭，不使用旧本地数据放行。

### 5.2 后台入口

`AppHeader` 的桌面用户菜单和移动用户菜单只有在 `user.role === 'ADMIN'` 时才显示“后台管理”。普通用户不渲染该入口。

隐藏菜单只是体验优化，不代替路由和后端权限校验。

### 5.3 路由守卫

新增职责单一的管理路由守卫，统一包裹 `/admin`、`/admin/users`、`/admin/questions` 和 `/admin/oss`：

- 认证尚未初始化：显示现有风格的加载状态，不渲染管理页面。
- 没有登录：使用 React Router `Navigate` 跳转 `/login`，并 `replace` 当前历史记录。
- 已登录但角色不是 `ADMIN`：跳转首页 `/`，并 `replace` 当前历史记录。
- 角色为 `ADMIN`：渲染现有 `AdminLayout` 和目标管理页面。
- 认证初始化失败：沿用全局认证错误和重试行为，不根据旧本地角色放行。

后台页面自身不重复实现角色判断，避免各页面规则不一致。

## 6. 请求与状态流

### 6.1 登录管理员

1. 用户提交邮箱和密码。
2. 后端验证密码并从数据库读取角色。
3. 响应包含 JWT 和 `role: ADMIN`。
4. 前端保存认证信息并显示后台入口。
5. 访问后台时路由守卫允许渲染，管理 API 再次从数据库验证管理员角色。

### 6.2 登录普通用户

1. 登录响应包含 `role: USER`。
2. 前端不显示后台入口。
3. 直接输入 `/admin` 被前端跳转首页。
4. 绕过前端直接调用管理 API 时，后端返回 HTTP 403。

### 6.3 角色或账号状态变化

管理员角色在数据库中被移除、账号被禁用或软删除后：

- 后续 `/auth/me` 返回当前状态对应结果；
- 后续管理 API 不再放行；
- 已签发的 JWT 不能保留管理权限。

## 7. 错误处理

- 无 Token、格式错误、签名错误、过期 Token、缺少载荷：HTTP 401，`未登录或登录已过期`。
- Token 对应用户不存在、已删除或已禁用：HTTP 401。
- 活动普通用户访问管理接口：HTTP 403，`需要管理员权限`。
- 数据库缺少 `role` 字段：属于部署顺序错误，后端启动或查询失败；通过“先迁移、后部署”规避，不在运行时静默降级。
- 前端收到管理 API 的 403：保留登录态，不把用户误判为退出登录；管理页面显示请求错误或由路由守卫离开后台。
- 前端收到 401：沿用现有认证失效处理，清除本地认证信息。

## 8. 测试与验收

### 8.1 后端自动测试

为管理员校验补充测试，至少覆盖：

- 活动 `ADMIN` 使用有效 Token，`requireAdmin()` 返回用户 ID。
- 活动 `USER` 使用有效 Token，抛出 `ForbiddenException`。
- 空角色和未知角色使用有效 Token，抛出 `ForbiddenException`。
- 无效或过期 Token 返回 401 路径。
- 已禁用、已删除或不存在的用户返回 401 路径。
- 管理控制器在调用业务服务前执行管理员校验。

### 8.2 构建检查

- Maven 测试和 `clean package` 成功。
- 前端 `npm run build` 成功。
- 不新增 TypeScript 构建错误。

### 8.3 数据库与接口验收

1. 迁移前备份数据库。
2. 执行迁移后验证 `admin@aisoftoj.local` 恰好一条记录为 `ADMIN`，其他现有用户为 `USER`。
3. 管理员登录响应和 `/auth/me` 均包含 `role: ADMIN`。
4. 普通用户登录响应和 `/auth/me` 均包含 `role: USER`。
5. 管理员访问 `GET /admin/dashboard` 返回 HTTP 200。
6. 普通用户分别访问管理端只读和写接口，均返回 HTTP 403，且数据没有变化。
7. 普通用户上传一个合法图片到 `POST /oss/upload` 返回 HTTP 403，且 `OssService` 未被调用、Bucket 没有新增对象。
8. 无 Token 访问管理接口和 `POST /oss/upload` 均返回 HTTP 401。
9. 管理员仍可正常使用用户管理、题库管理和 OSS 管理页面，并能通过 `POST /oss/upload` 上传合法图片。
10. 普通用户在桌面和移动菜单均看不到后台入口，直接访问四个管理路由均不能看到管理内容。
11. 篡改 `localStorage` 中的角色只能影响本地展示，不能绕过后端 403。

## 9. 部署顺序

1. 备份生产数据库。
2. 执行 `db_migrations/20260715_add_user_role.sql`。
3. 验证管理员数量和邮箱。
4. 部署后端并完成 200/401/403 接口检查。
5. 部署前端并检查菜单和管理路由。
6. 修改 `admin@aisoftoj.local` 的初始密码后再开放公网。

## 10. 非目标

本轮不包含：

- 多角色、多权限或用户与角色多对多关系；
- 后台角色授予、降权或角色管理界面；
- 在 JWT 中保存角色；
- Refresh Token、Cookie 化认证或整体认证框架重写；
- 管理员操作审计日志；
- 登录限流、验证码和 OSS 配额；
- 管理后台视觉重设计；
- 与管理员角色无关的业务重构。

## 11. 完成标准

- 数据库拥有默认拒绝的用户角色字段，现有管理员账号被唯一、明确地授予 `ADMIN`。
- 后端所有 `/admin/**` 及管理用途的 `POST /oss/upload` 对普通用户返回真实 HTTP 403，对无效身份返回 HTTP 401。
- 管理权限实时依据数据库角色，客户端和旧 JWT 无法提升或保留权限。
- 前端只向管理员展示后台入口，并统一保护所有后台路由。
- 现有普通用户注册、登录、刷题和个人数据流程不回归。
- 后端自动测试、前后端生产构建和真实接口验收全部通过。
