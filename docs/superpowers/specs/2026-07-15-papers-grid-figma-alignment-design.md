# 题库展示与 Figma 对齐设计

## 背景

当前 `/papers` 页面已经接入真实 API、游客浏览、登录校验、练习会话续答和错误状态，但试卷列表仍使用两列大卡片与灰色统计面板。用户提供的 [Figma Make 参考稿](https://www.figma.com/make/SWQfsGunfx5VC8yrxALnDr/aisoft?p=f&t=0stpUXszCCjic8bu-0) 采用更紧凑的三列卡片、线性指标和明确的练习进度，因此同一桌面视口下的信息密度和层级存在明显差异。

本设计主要调整科目筛选区和试卷列表，并为现有 `/paper/list` 响应补充卡片所需的真实完成题数。页头、页面标题、功能 Tab、认证流程、答题模式弹窗以及开始或继续会话的业务行为保持不变。

## 目标

- 在桌面宽度下将试卷列表调整为三列，视觉密度接近 Figma 参考稿。
- 用线性指标替代当前灰色统计面板，依次展示题目数量、练习次数和练习进度。
- `/paper/list` 在已认证请求中返回当前用户的真实完成题数，不增加前端逐卡请求。
- 所有试卷卡片都展示进度条，包括游客或未开始的 `0/总题数`、进行中的真实完成题数和已完成的 `总题数/总题数`。
- 保留未开始、进行中、已完成以及游客状态各自的操作按钮和业务行为。
- 保留真实 API 为唯一数据来源，不为了匹配设计而添加不存在的科目、题型或浏览量。
- 同步调整加载骨架屏，使加载态和完成态不会发生明显布局跳变。

## 非目标

- 不调整 `AppHeader`、页面标题和三个功能 Tab。
- 不新增接口，不修改数据库结构或练习会话流程。
- 不将 `practiceCount` 重命名或伪装成浏览次数。
- 不直接修改 `src/components/ui/` 下的 Shadcn 组件。
- 不重新设计刷题记录、错题分析、模式选择弹窗或错误空状态。

## 设计方案

### 筛选区

筛选区继续使用单个白色容器承载科目与题型两组控件，并保留两组之间的细分隔线。视觉调整以贴近参考稿的间距和按钮密度为主，不改变选择逻辑。

- 科目和题型选项继续从试卷 API 返回值推导；没有对应试卷的数据不显示虚假选项。
- 选中科目使用蓝底白字，选中题型使用深色底白字。
- 未选中项使用白底、浅色边框和中性色文字。
- 按钮继续自动换行，以适应可变数量的选项和较窄视口。

### 列表标题

列表标题保持“科目 · 题型”和右侧试卷总数的结构。桌面横向排列，窄屏允许上下排列，避免标题截断或计数徽标挤压。

### 响应式网格

- 手机：单列。
- 中等宽度：两列。
- `xl` 及以上：三列。
- 桌面卡片间距使用约 24px，卡片之间不再因两列布局显得过宽。

加载骨架屏使用相同断点和列数。

### 试卷卡片

卡片采用白色背景、浅色描边、克制阴影和约 14–16px 圆角。每张卡片使用纵向弹性布局，保证底部更新时间和操作按钮对齐。桌面最小高度约 300px，使用内联 `minHeight`，避免项目中 Tailwind 任意值的构建不稳定问题。

卡片内容由上到下为：

1. 标题、科目名和右上角题型徽标。
2. 题目数量，左右对齐。
3. 练习次数，左右对齐。
4. 练习进度数值，左右对齐。
5. 深色进度条。
6. 分隔线、更新时间和状态操作按钮。

移除当前灰色统计面板与独立“进行中”提示条。进行状态已经由进度和操作按钮共同表达，不再重复占用卡片空间。

### 数据与状态规则

卡片只使用现有 `ExamPaper` 字段：

- `questionCount`：题目总数。
- `practiceCount`：练习次数，缺失时显示 `0`。
- `completedCount`：当前用户在用于续答的进行中会话里的已答题数；游客响应可以缺失。
- `status`：决定操作按钮与完成进度的回退值。
- `lastUpdated`：继续使用现有日期格式化函数。

状态是进度展示的最高优先级，展示前按以下顺序归一化：

- `status === 'completed'` 时始终使用 `questionCount`，即使响应里存在更小的 `completedCount`。
- `status === 'in_progress'` 时使用 `completedCount`，缺失时回退为 `0`，并限制在 `0..questionCount` 范围内。
- 游客或 `status === 'not_started'` 时使用 `0`。
- `questionCount` 为 `0` 时进度百分比为 `0`，避免除零。

操作行为保持现状：

- 游客点击“开始刷题”进入登录流程。
- 未开始用户点击“开始刷题”打开模式选择。
- 进行中用户点击“继续刷题”直接续接当前练习会话。
- 已完成用户点击“重新刷题”打开模式选择。

## 组件边界

前端主要修改 `aisoftoj-front/src/components/PapersPage.tsx`：

- 保留 API 请求、筛选状态、认证逻辑和弹窗逻辑。
- 在文件内新增小型纯计算辅助函数，用于归一化完成题数和计算进度百分比。
- 重排 `PapersSkeleton` 和试卷卡片 JSX。

不抽取新的公共卡片组件，因为该卡片目前只由题库页使用，抽取会增加接口和文件复杂度而没有复用收益。

后端修改保持在现有试卷目录边界内：

- `PaperDTO` 增加可选的 `completedCount` 字段。
- `PaperServiceImpl` 复用当前已经查询出的用户会话，不新增数据库查询。
- 构建进行中会话映射时，`doingSessionId` 和 `completedCount` 必须取自同一个 `PracticeSession`，避免续答会话与展示进度不一致。
- 已完成试卷返回 `questionTotal` 作为 `completedCount`；未开始返回 `0`；游客继续不输出个人字段。

前端 `fetchPapers()` 已兼容 `completedCount`，继续将游客缺失值归一为 `0`，无需引入额外请求。

## 数据流

1. 后端继续一次性查询公开试卷；已认证请求继续一次性查询当前用户会话。
2. 后端从同一个进行中会话生成 `doingSessionId` 与 `completedCount`，并随 `/paper/list` 返回。
3. 页面继续通过 `fetchPapers()` 获取试卷目录，不增加逐卡请求。
4. `subjectOptions` 和 `categoryOptions` 继续从返回试卷推导。
5. `filteredPapers` 按当前科目和题型过滤，并按年份和月份排序。
6. 每张卡片根据状态归一化完成题数，并据此计算展示文本和进度百分比。
7. 操作按钮继续调用现有 `onShowAuth`、`openModeDialog` 或 `onStartPaper` 回调。

本次没有新的接口、额外请求、缓存或持久化状态。

## 错误与边界处理

- 登录初始化、试卷加载失败、空列表和游客提示保持现状。
- 筛选后没有匹配试卷时，应保留页面结构并显示现有或轻量的空结果反馈，不产生运行时错误。
- 负数、超出总题数或缺失的完成题数会被归一化。
- 后端进行中会话的 `answeredCount` 缺失时返回 `0`；已完成状态始终覆盖异常计数并返回总题数。
- 很长的科目名继续截断，题型徽标禁止收缩。
- `prefers-reduced-motion` 下不引入新的位移动画；沿用当前卡片的 reduced-motion 处理。

## 验证方案

### 自动验证

- 运行 `npm run build`，确认 TypeScript、Vite 和 Tailwind 构建通过。
- 运行后端现有 Maven 测试，验证 `PaperDTO` 序列化与试卷目录逻辑没有回归。
- 若项目现有测试环境可用，补充或运行针对后端会话映射和前端进度归一化函数的边界测试；不为本次调整额外引入测试框架。

### 视觉验证

在真实页面分别检查：

- 桌面宽度：三列卡片、约 24px 间距、底部操作对齐。
- 平板宽度：两列卡片，无文字或按钮溢出。
- 手机宽度：单列卡片，筛选按钮正常换行。
- 未开始、进行中、已完成与游客四种状态。
- 加载骨架屏与完成态列数一致。
- 题目总数为零、完成数缺失和完成数越界的进度显示。

最终截图与用户提供的 Figma 参考图对比，重点核对列数、卡片纵向节奏、指标层级、进度条和底部操作区。

## 已批准决策

- 采用方案 A“Figma 结构对齐”。
- 前端视觉调整范围仅限筛选区和试卷列表。
- 使用“练习次数”，不展示不存在的浏览次数。
- 所有卡片显示进度。
- 状态按钮继续遵循现有业务逻辑。
- 为保证进行中进度真实，允许 `/paper/list` 和 `PaperDTO` 增加 `completedCount`，但不新增接口或数据库查询。
- 已完成状态优先于异常完成计数，始终显示满进度。
