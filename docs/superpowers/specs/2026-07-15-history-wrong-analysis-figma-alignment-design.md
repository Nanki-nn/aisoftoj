# 刷题记录与错题分析 Figma 对齐设计

## 背景

当前 `/practice-history` 与 `/wrong-questions` 已接入真实分页 API，并提供继续答题、查看记录、查看结果、查看原题和移除等操作，但两页仍采用传统表格与独立返回栏，和 `/papers` 以及用户提供的 Figma 参考稿在页面标题、Tab、统计摘要、列表密度和背景风格上不一致。

参考设计来自同一 [Figma Make 文件](https://www.figma.com/make/SWQfsGunfx5VC8yrxALnDr/aisoft?p=f&t=0stpUXszCCjic8bu-0)。Figma Make 不支持 MCP 截图接口，因此以用户提供的刷题记录、错题分析截图和 Make 源码共同作为视觉基线。

## 已确认原则

- 保留独立路由、真实分页和现有有效操作，不为了还原 mock 而删除业务能力。
- 三个页面统一为“刷真题标题 + 三个 Tab + 页面内容”的工作区结构。
- 刷题记录和错题分析增加四张统计卡片。
- 统计值必须来自全量数据，不能用当前页记录临时计算。
- 用户要求后续不再停下来确认，按本设计自主完成并验证。

## 方案比较

### A. 仅美化现有表格

改动最小，但仍保留宽表格、面包屑返回栏和缺少统计摘要的问题，与参考稿差距明显。

### B. 真实能力优先的 Figma 对齐版（采用）

抽取共享工作区标题与 Tab，增加真实全量统计，把桌面表格改为响应式列表行，同时保留分页和所有有效操作。视觉接近 Figma，业务风险可控。

### C. 完全合并到 `PapersPage`

最接近 Figma Make 的单组件 mock，但会把三个独立 API 页面和大量状态集中到一个组件，破坏路由可访问性并增加维护复杂度，不采用。

## 前端结构

### 共享工作区头部

新增 `PapersWorkspaceHeader`，由 `/papers`、`/practice-history` 和 `/wrong-questions` 共用：

- 展示“历年真题题库 / 刷真题”标题和蓝色文档图标。
- 展示试卷列表、刷题记录、错题分析三个路由 Tab。
- 当前 Tab 使用蓝底白字，非当前 Tab 使用中性色。
- 已登录时展示刷题记录与错题总数；游客在受保护 Tab 上跳转到登录页。
- 桌面为紧凑横向导航，窄屏允许横向滚动，不隐藏任何 Tab。

`PapersPage` 继续自行渲染 `AppHeader`；另两页继续由现有 `AppShell` 提供 `AppHeader`。共享组件只负责工作区标题和 Tab，避免重复顶栏。

### 刷题记录页

页面顺序：共享头部、四张统计卡片、练习记录列表。

统计卡片：

- 总练习次数。
- 进行中。
- 已完成。
- 累计答题。

列表从宽表格改为响应式行：

- 左侧为题库名称、题型和考试/练习模式徽标。
- 次级信息展示创建时间和 `已答/总题数`。
- 右侧展示进度条、状态徽标和操作。
- 进行中保留“继续”；已完成保留“查看”和“结果”。
- 列表头保留每页条数选择，列表底部保留总数和翻页。
- 手机端右侧区域移到主信息下方，按钮保持可触达。

### 错题分析页

页面顺序：共享头部、四张统计卡片、错题列表。

统计卡片：

- 错题总数。
- 必须掌握：沿用前端现有重要级别映射，统计当前标签为“必须掌握”的 `high` 级记录。
- 高频错题：`error_count >= 2`。
- 涉及题库：不同 `paper_name` 数量。

列表从宽表格改为响应式行：

- 左侧为题目名称、题型和重要级别徽标。
- 次级信息展示所属题库和更新时间。
- 右侧突出错误次数，并保留“查看原题”和“移除”。
- 列表按错误次数降序、最近错误时间降序排列，与页面说明一致。
- 保留每页条数选择、总数和翻页。
- 移除当前没有事件处理的“生成练习”假按钮，不新增未实现功能。

## 后端与数据契约

### 带摘要的分页 DTO

新增通用 `PageWithSummaryDTO<T, S>`：

- `records`
- `total`
- `page`
- `pageSize`
- `summary`

只用于两个用户统计接口，不修改后台管理等现有 `PageDTO` 响应。

### 刷题记录摘要

新增 `PracticeHistorySummaryDTO`：

- `totalCount`
- `inProgressCount`
- `completedCount`
- `answeredCount`

`PracticeSessionMapper` 使用一条聚合 SQL 对当前用户、未删除会话统计。分页列表查询保持不变。

### 错题摘要

新增 `WrongQuestionSummaryDTO`：

- `totalCount`
- `masterCount`
- `frequentCount`
- `paperCount`

`UserWrongQuestionStatMapper` 使用一条聚合 SQL 计算摘要，并把分页列表排序调整为 `error_count DESC, last_wrong_time DESC, id DESC`。

### 前端类型

扩展分页结果类型以支持可选泛型摘要，并分别定义刷题记录、错题分析摘要类型。两个 API 函数返回强类型摘要。

为兼容尚未重启的旧后端，摘要字段缺失时使用唯一且不误导的回退规则：总数卡片使用分页 `total`；`inProgressCount`、`completedCount`、`answeredCount`、`masterCount`、`frequentCount` 和 `paperCount` 统一显示 `--`，不从当前页聚合，也不显示可能被误解为真实统计的 `0`。新后端启用后以服务端摘要为准。

### Tab 计数

- 当前页直接使用自身分页响应的 `total`。
- 刷题记录页额外请求错题接口 `pageSize=1` 获取错题总数。
- 错题分析页额外请求刷题记录接口 `pageSize=1` 获取记录总数。
- 补充计数请求失败不阻断主列表，只隐藏或显示占位计数。

## 状态、错误和边界

- 主列表加载、错误、空状态分别使用与新页面一致的容器和文案。
- 统计摘要加载时使用骨架块，避免页面跳动。
- 单条记录的答题数和错误次数缺失时回退为 `0`；摘要字段缺失时按上一节显示 `--`。
- 进度百分比限制在 `0..100`，总题数为 `0` 时为 `0`。
- 很长的题库名、题目名和所属题库在桌面截断，在移动端允许合理换行。
- 翻页或切换每页条数时保留现有请求行为。
- 错题“移除”维持当前页面内移除行为，只隐藏当前页对应行，不调整服务端 `total` 或摘要；重新加载后记录会恢复。本次不将其描述为持久删除，也不新增删除 API。

## 视觉规则

- 页面背景沿用 `/papers` 的浅蓝到浅灰渐变。
- 主容器沿用 `max-w-7xl`、桌面约 32–48px 垂直节奏。
- 统计区桌面四列、平板两列、手机两列或单列，卡片使用轻描边、白底和克制阴影。
- 列表容器使用约 16px 圆角；行高约 88–104px。
- 颜色语义：蓝色为总量，琥珀为进行中，绿色为已完成，红色为错误，紫色为累计或涉及范围。
- 使用现有 `lucide-react` 图标和 Shadcn 组件，不修改 `components/ui/`。

## 测试与验收

### 后端

- 为 `UserStatsController` 或相关映射逻辑补充测试，验证两个接口返回分页字段与摘要。
- 验证刷题记录的进行中、已完成和累计答题聚合。
- 验证错题总数、必须掌握、高频错题、涉及题库聚合。
- 验证错题分页排序与摘要不受页大小影响。
- 运行后端全量 Maven 测试。

### 前端

- 运行 `npm run build`。
- 验证三个 Tab 路由和激活态。
- 验证继续、查看、结果、查看原题和移除操作仍可触发。
- 验证 1440px、900px 和 390px 三档布局，无横向溢出。
- 验证加载、空列表、接口错误和旧后端缺少摘要的兼容状态。
- 验证旧后端场景只有总数可用，其余六个摘要指标显示 `--`，不会从当前页计算。
- 对照用户提供的两张参考图检查统计卡片、列表密度、徽标、进度和错误次数层级。

## 非目标

- 不合并三个路由。
- 不新增错题生成练习功能。
- 不新增错题持久删除 API。
- 不修改数据库表结构。
- 不修改答题会话和查看原题的既有业务流程。
