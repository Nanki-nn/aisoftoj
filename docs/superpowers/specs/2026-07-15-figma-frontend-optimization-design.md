# 知构软考首批前端优化设计

日期：2026-07-15

状态：已确认，待实施

范围：首页、全局导航、真题列表

## 1. 背景与目标

本轮优化参考 Figma Make 文件 `aisoft` 的首页和真题列表页，在不改变现有业务接口、路由及答题流程的前提下，统一首批高曝光页面的视觉语言与信息层级。

固定设计基准：[Figma Make `aisoft`](https://www.figma.com/make/SWQfsGunfx5VC8yrxALnDr/aisoft?p=f&t=psDu1cudT29SZrXY-0)，版本 28，页面路径 `/home` 与 `/papers`。桌面视觉基准为 1366×768；响应式另外按第 8 节的四个宽度验收。Figma MCP 在当前环境未配置，因此实施以该版本的可见预览和本规格的唯一化决策为准。

目标：

- 首屏清楚表达“按路径备考”的产品价值，并提供明确的刷题入口。
- 全局导航在桌面端与移动端均可用，登录态和现有菜单功能保持不变。
- 真题页按照“功能入口、筛选、试卷列表”的顺序组织信息，降低选择成本。
- 真实 API 的加载、失败和空数据状态有完整反馈；开始会话仍复用现有错误处理。
- 后续页面可复用本轮形成的颜色、间距、圆角、阴影和组件模式。

## 2. 设计原则

采用“高保真参考 + 生产化修正”：

- 高保真参考 Figma 的浅蓝页面背景、白色卡片、蓝色主操作、三条路线识别色和宽松留白。
- 不照搬 Figma Make 中的 mock 数据或重复页面；试卷、进度状态、用户统计等业务数据来自项目现有状态和 API。品牌说明、备考方法等运营文案允许作为明确的静态内容存在，但不得伪装成实时统计。
- Figma 中“刷题记录、错题分析”的页签复用现有路由 `/practice-history` 和 `/wrong-questions`。
- 不直接修改 `src/components/ui/` 下的 Shadcn 组件。
- 优先使用标准 Tailwind 类；确有设计色无法稳定表达时再使用 CSS 变量或行内样式。

## 3. 信息架构

### 3.1 全局导航

桌面端从左到右为：

1. 品牌标识“知构软考”，点击返回首页。
2. 主导航：打基础、刷真题、论文冲刺。
3. 考试倒计时。
4. GitHub Star 入口。
5. 登录/注册按钮，或已登录用户头像与菜单。

当前路由统一使用蓝色实底、白色文字高亮，避免仅依赖文字颜色。导航固定在页面顶部，滚动时保留轻微边框与背景模糊。

小于 1024px 时保留品牌、倒计时简版和菜单按钮；主导航、GitHub、登录态操作收进基于现有 Shadcn `Sheet` 的侧滑面板。打开后焦点进入面板，Tab 焦点不能逃逸，关闭按钮、点击导航项或 Escape 均可关闭，关闭后焦点回到菜单按钮。

### 3.2 首页

首页按以下顺序组织：

1. **首屏 Hero**：主标题“软考备考，从路径开始。”，一句精简说明，主按钮“直接刷真题”进入 `/papers`，次按钮“先看备考路径”滚动到本页 `#learning-paths`。
2. **三条学习路线**：打基础、刷真题、论文冲刺三张卡片，分别使用蓝、绿、紫作为识别色；卡片使用链接语义并依次进入 `/foundation`、`/papers`、`/essay-sprint`。
3. **作者经历**：保留真实备考路径与三科通过结果，压缩装饰与文案宽度，作为信任证明而非第二个 Hero。
4. **方法说明**：解释三阶段为何这样安排，保持三列结构。
5. **产品能力摘要**：只展示可由产品定义直接证明的固定内容——“3 条学习路线”“2 个月冲刺框架”“AI 辅助论文批改”。删除无数据来源的“85% 通过率”和“5,000+ 题量”宣称。

桌面首屏应在常见 1366×768 视口内同时露出 Hero 主操作和三条路线卡片上缘。移动端所有内容改为单列，按钮在窄屏下可纵向排列。

### 3.3 真题页

页面按以下顺序组织：

1. 页面标题“刷真题”。
2. 功能导航：试卷列表、刷题记录、错题分析。它们是跨路由链接而非 ARIA tab；后两项分别显示 `fetchPracticeHistory({ pageSize: 1 }).total` 的会话总数与 `fetchWrongQuestions().total` 的错题条目总数。
3. 白色筛选卡：科目筛选、题型分类两组胶囊按钮。
4. 当前筛选标题和试卷数量。
5. 响应式试卷卡片列表。

试卷卡内容包括：

- 年份与月份标题；
- 科目与题型标签；
- 题目数量与练习次数；练习次数直接展示 `paper.practiceCount`，其值来自当前接口返回；
- 进行中状态标签；本轮不显示数值进度，因为现有接口的 `completedCount/progress` 尚不是可靠的当前会话进度；
- 更新时间；
- 根据状态显示“开始刷题”“继续刷题”或“重新刷题”。

试卷卡在 768px 及以上使用两列，小于 768px 使用单列。筛选项允许横向换行，不出现横向页面滚动。

## 4. 视觉系统

### 4.1 色彩

- 页面背景：`slate-50` 到浅蓝色的轻渐变，真题内容区以 `blue-50/40` 为主。
- 主色：`blue-600`，悬停使用 `blue-700`。
- 主文字：`slate-900`；正文：`slate-600`；辅助文字：`slate-500`。
- 路线识别：打基础 `blue-*`、刷真题 `emerald-*`、论文冲刺 `violet-*`。
- 倒计时：`amber-50` 背景配 `amber-800` 文本。
- 错误、空状态、成功状态使用标准 `red-*`、`slate-*`、`emerald-*`。

### 4.2 版式与组件外观

- 内容最大宽度统一为 `max-w-7xl`，左右边距在移动端为 16px，桌面端为 24–32px。
- Hero 标题桌面端约 48–56px，移动端约 36–40px，控制在两行以内。
- 主要卡片圆角使用 16–20px，按钮使用 10–12px；避免每个区域使用不同圆角。
- 卡片默认使用细边框与轻阴影，悬停只做小幅上移、边框加深和阴影增强。
- 交互动画控制在 150–250ms，并遵守 `prefers-reduced-motion`。

## 5. 组件边界

### 5.1 `AppHeader`

职责：品牌、主导航、倒计时、GitHub 入口、登录状态和移动端菜单。

依赖：React Router、`useAuth`、现有 Avatar/Dropdown/Button 组件。允许 `useAuth` 新增只表示首次认证检查完成的 `isAuthInitialized` 状态；该状态不改变 Token、用户数据或登录规则。

不负责：页面内容、后台权限判断或用户数据请求。

### 5.2 `LearningLanding`

职责：首页各内容区的结构、文案与导航动作。

建议在文件内部或独立轻量组件中拆分 `HeroSection`、`LearningPathCard`、`AuthorStory` 和 `CapabilityStrip`，避免主组件继续增长。

不新增服务端请求；点击动作继续通过 React Router 导航。

### 5.3 `PapersPage`

职责：加载真题和用户统计、管理筛选状态、渲染功能页签和试卷卡、打开练习模式选择。

保持现有 `fetchPapers`、`fetchPracticeHistory`、`fetchWrongQuestions` 调用和 `onStartPaper` 回调。允许抽取纯展示的 `PaperCard`、`FilterGroup` 和 `PapersPageNav`，但不改变 API 协议。`ExamPaper` 类型迁移到 `src/types/exam.ts` 或独立类型文件；`PapersPage` 不再从 `src/data/examPapers.ts` 导入筛选常量或业务类型。

## 6. 数据流与状态

### 6.1 首页与导航

- 路由状态决定当前主导航高亮。
- `useAuth` 决定显示登录按钮或用户头像菜单。
- 倒计时由 `AppHeader.tsx` 中唯一常量 `2026-05-23T00:00:00+08:00` 计算；按该固定时区比较，当前时间达到或超过考试日后统一显示“本期考试已结束”，不显示 `0 天`，也不推测下一考试日期。

### 6.2 真题页

1. `useAuth` 在首次 `checkAuthStatus()` 的同步/异步分支全部结束后将 `isAuthInitialized` 设为 `true`。`PapersPage` 仅在该值为 `true` 后判断是否发请求。未登录访问 `/papers` 时不发起试卷、历史或错题请求，在页面主体显示“登录后查看真题与学习进度”及“去登录”按钮；按钮进入 `/login`。本轮不修改登录后的回跳逻辑。
2. 已登录后并行请求试卷、`fetchPracticeHistory({ pageSize: 1 })` 和错题统计。试卷请求控制主要加载与错误状态，统计请求失败不阻断试卷列表，对应计数固定显示 `--`。
3. 科目选项由 `fetchPapers()` 返回值中的非空 `subject` 去重生成；已知科目按“系统架构设计师、系统分析师、软件设计师、网络工程师、数据库系统工程师、信息系统项目管理师”排列，未知科目使用 `localeCompare('zh-CN')` 后追加。题型选项由当前科目下的非空 `category` 去重生成，按“综合知识、案例分析、论文”排列，未知题型同样按中文名称追加。
4. 初次加载选择第一个真实科目及该科目的第一个真实题型，不提供“全部”项，以匹配 Figma。科目切换后，如果当前题型在新科目下不存在，立即重置为该科目的第一个真实题型。没有任何试卷时不设置选择值，进入全局空状态。
5. 点击试卷操作时复用现有模式选择和开始/继续会话逻辑。现有 `completedCount/progress` 不参与展示；`status === 'in_progress'` 只显示“进行中”和“继续刷题”。
6. 登录令牌在请求期间失效时，试卷主请求按失败态展示后端消息和“去登录”按钮；不自动开放匿名接口，也不在本轮重写鉴权流程。

## 7. 异常与边界状态

- **加载中**：统一使用与最终布局一致的筛选卡骨架和四张试卷卡骨架，避免页面跳动。
- **试卷请求失败**：显示错误原因和“重新加载”按钮；点击后重新执行试卷、历史总数、错题总数三项页面请求，使网络恢复后的计数同步更新。
- **无试卷**：接口整体为空时显示“暂时还没有可用试卷”。由于题型选项只从当前科目的真实试卷生成，不设计不可到达的“筛选组合为空”状态。
- **统计请求失败**：不影响试卷主流程，不弹阻塞式提示，对应导航计数显示 `--`。
- **开始会话失败**：保留 `App.tsx` 现有 `alert` 行为和错误内容；Toast 改造、请求中禁用和失败后恢复模式弹窗明确不属于本轮完成标准。
- **长科目名**：筛选按钮允许自适应宽度与换行，卡片标题不得溢出。
- **考试日期过期**：固定显示“本期考试已结束”。

## 8. 可访问性与响应式

- 所有可点击卡片使用真实按钮或链接语义，不能只依赖 `div onClick`。
- 图标按钮提供中文 `aria-label`；装饰图标对读屏隐藏。
- 焦点样式清晰，颜色对比达到 WCAG AA 的常规文本要求。
- 当前主导航和功能导航使用真实链接及 `aria-current="page"`，不使用 `role="tab"`。
- 移动菜单复用 Shadcn `Sheet`，模式选择复用 Shadcn `Dialog`；不修改其 `ui/` 实现。两者打开后焦点进入面板、Tab 不逃逸、Escape 关闭、关闭后焦点返回触发按钮。
- 360px：移动导航、单列 Hero 按钮、单列路线卡和试卷卡；768px：移动导航、三列路线卡、两列试卷卡；1024px 与 1366px：桌面导航、三列路线卡、两列试卷卡。
- 四个验收宽度均需检查可见焦点、WCAG AA 常规文字对比度、`prefers-reduced-motion` 下无非必要位移动画，以及页面无横向滚动。

## 9. 测试与验收

### 9.1 自动检查

- `npm run build` 必须通过。
- 若项目补充类型检查或 lint 脚本，必须无新增错误。
- 浏览器控制台不得出现新增运行时错误或资源 404。

### 9.2 手工流程

1. 未登录访问首页，验证“直接刷真题”进入 `/papers`、“先看备考路径”滚动到 `#learning-paths`，三条路线依次进入 `/foundation`、`/papers`、`/essay-sprint`。
2. 已登录时头像菜单、个人中心、后台入口和退出行为不回归。
3. 在四个验收宽度验证导航布局；移动 `Sheet` 和模式 `Dialog` 均完成打开后聚焦、Tab 圈定、Escape 关闭、关闭后焦点恢复。
4. 未登录直接访问 `/papers` 时不产生三类业务请求，只显示登录引导；已登录后真题列表加载成功，动态科目与题型组合筛选及切换重置正确。
5. 开始、继续、重新刷题三种状态仍调用原有流程。
6. 刷题记录与错题分析页签进入现有路由。
7. 完成一条答题回归链：登录 → 打开试卷 → 分别验证练习/考试模式入口 → 作答并保存 → 刷新后继续 → 交卷 → 查看结果 → 重新刷题；开始会话请求失败时现有错误提示仍出现，且已有的当前会话状态不被前端清空。服务端幂等与网络不确定状态下的重复会话不属于本轮保证。
8. 分别复现主请求失败、统计失败、整体空数据、超长科目名和过期倒计时，确认唯一文案与布局；成功场景中的练习次数必须与 `paper.practiceCount` 一致。主请求失败可通过临时将 `VITE_API_BASE_URL` 指向未监听端口复现；其他状态使用浏览器请求替身或独立测试夹具，不把测试数据写入生产代码。
9. 对照固定的 Figma 版本 28 `/home` 与 `/papers` 预览，检查首页和真题页的主色、间距、卡片层级和主操作位置。

## 10. 非目标

本轮不包含：

- 登录、个人中心、答题页、结果页、错题页和论文页的全面重设计；
- 后端接口、数据库或鉴权逻辑调整；
- 新增统计 API；
- 修复后端试卷进度口径或在本轮展示数值进度；
- 重构开始会话的 `alert`、按钮禁用和弹窗错误恢复；
- 新增图标库或替换 Shadcn 基础组件；
- 将 Figma Make 的 mock 数据复制进生产代码。

## 11. 完成标准

- 首页、导航和真题列表形成一致的视觉系统，整体观感与 Figma 参考稿匹配。
- 现有真实数据、登录态、动态筛选和完整答题流程无回归；页面不再使用 mock 筛选常量，也不展示不可靠的数值进度。
- 360px 到 1366px 的关键宽度无横向滚动、内容遮挡或无法操作的控件。
- 加载、失败、空数据和已过期倒计时都有可理解的展示。
- 生产构建成功，并完成浏览器截图对照验证。
