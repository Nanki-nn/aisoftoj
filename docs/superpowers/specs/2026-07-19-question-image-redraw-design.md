# 题库图片统一重绘与 OSS 迁移设计

## 1. 目标

将当前本地数据库中正在被题目引用的图片全部重新绘制为原创、无水印、统一黑白风格的技术图表。在不改变题意和正确答案的前提下，允许优化排版、对齐、留白与信息层级；不得把标准答案或能够直接推导出填空答案的新增提示画入图片。

本设计分两个阶段执行：

1. 完成全部图片的 SVG 重绘、PNG 导出和总览验收；不修改数据库。
2. 用户验收后上传 OSS，备份并批量替换数据库中的旧图片 URL。

## 2. 精确范围

以当前运行中的本地 `aisoftoj.question` 表为唯一数据源，过滤 `is_deleted = 0`，检查 `intro`、`options`、`answer`、`analysis` 四个字段中的 Markdown 图片和 HTML 图片引用。

精确盘点结果：

- 有效题目：1163 道。
- 包含图片的题目：32 道。
- 图片 URL 引用次数：33 次。
- 唯一图片 URL：28 个。
- 系统架构设计师：25 张唯一图片。
- 系统分析师：3 张唯一图片。
- 两张图片分别被 4 道和 3 道连续小题复用；只重绘和上传一次。
- 一道题包含两张不同图片，两张都需要独立重绘。

普通文章链接、用户头像、README 图片、指南图片和历史导出目录中尚未导入数据库的图片不在本次范围内。

## 3. 方案选择

### 3.1 采用方案：确定性 SVG 重绘

每张图使用可编辑 SVG 表达，统一字体、颜色、线宽、箭头、表格、节点和留白，再从 SVG 导出高分辨率 PNG。

选择原因：

- 中文文字、编号、连线和拓扑关系可以精确控制。
- 可以稳定复现题目中的空白位置和编号占位。
- 统一风格后仍可针对复杂图表单独调整布局。
- SVG 可作为长期可维护源文件，PNG 可兼容现有 Markdown/HTML 图片展示。

### 3.2 不采用的方案

- AI 看图生成：容易产生中文错字、编号漂移、错误连线和答案泄露。
- 原图修补或像素级水印擦除：容易残留痕迹，无法统一视觉风格，也不利于后续维护。

AI 仅可辅助理解原图，不作为最终图中文字和结构的生成来源。

## 4. 内容重建规则

每张图片的内容由三类信息共同确定：

1. 原图用于识别整体结构、节点位置、箭头方向和表格形态。
2. 题目的 `intro` 用于确认题目要求、候选项、编号与空白位置。
3. `answer` 和 `analysis` 仅用于内部校验“不能画出的内容”，不得复制到最终图片。

规则如下：

- 保留题意所需的标题、实体名称、关系、箭头方向、编号、边界、表头和图例。
- 填空处继续使用 `(1)`、`(2)`、`a`、`b` 等原题占位，不填答案。
- 如果原图本身已经用红字、标注或高亮展示答案，最终图必须恢复为空白占位或原题编号。
- 删除来源网站名称、公众号、二维码、Logo、斜向水印、推广语和与答题无关的装饰。
- 不添加新的提示性颜色、说明文字或视觉强调。
- 当原图文字与题干明显冲突时，记录为人工复核项，不自行猜测。

## 5. 统一视觉规范

- 画布：白色背景，内容自适应宽高，四周保留至少 48 px 安全边距。
- 主色：`#111111`。
- 次级文字与辅助线：`#555555`、`#888888`。
- 禁止：彩色填充、渐变、阴影、纹理、Logo、水印和二维码。
- 字体：优先 `Noto Sans CJK SC`，回退到 `PingFang SC`、`Microsoft YaHei`、sans-serif。
- 正文字号：导出后等效 24–30 px；小型表格不得低于 20 px。
- 线宽：主结构 2.5 px，辅助线 1.5–2 px。
- 节点：圆角矩形、圆形、菱形和数据库圆柱采用统一描边风格。
- 箭头：统一箭头尺寸和端点留白，避免箭头穿过文字或节点。
- 表格：统一单线边框、居中表头、内容按语义左对齐或居中。
- 图内不得出现最终答案专用的红色、绿色或高亮状态。

## 6. 文件组织与命名

重绘产物保存在工作区的本地素材目录：

```text
assets/question-images-redrawn/
├── manifest.json
├── svg/
│   └── q-<question-id>-<field>-<source-hash>.svg
├── png/
│   └── q-<question-id>-<field>-<source-hash>.png
└── qa/
    ├── contact-sheet.png
    └── review-report.md
```

其中 `source-hash` 使用原图片 URL 的短 SHA-1，确保同一原图在多道题中复用同一份产物。`manifest.json` 保存旧 URL、题目 ID、字段、复用关系、SVG/PNG 路径、审核状态和未来 OSS URL。

当前 `assets/` 已被 `.gitignore` 忽略，因此批量二进制产物不会意外进入 Git。设计文档和后续迁移脚本单独管理。

## 7. 重绘流水线

### 7.1 清单生成

- 从数据库重新提取图片引用。
- 规范化 URL，并按完整 URL 去重。
- 记录每个 URL 对应的题目 ID、题目名称、字段和出现位置。
- 下载原图到本地参考目录，校验 HTTP 状态、文件类型和尺寸。

### 7.2 单图重绘

- 检查原图与题干，列出必须保留的文字和关系。
- 明确所有不可出现的答案内容。
- 使用共享 SVG 组件绘制节点、箭头、表格、时序图、数据流图等。
- 对复杂图表允许改变布局，但不能改变语义或连线方向。
- 导出 SVG 和 2 倍尺寸 PNG。

### 7.3 自动检查

- SVG/XML 可解析。
- PNG 能正常解码，尺寸和文件大小合理。
- 四角和主要背景为白色，不存在透明异常。
- 文件中不包含已知来源域名、水印文案、公众号或二维码文本。
- 图中文字清单与预期标签清单一致。
- 标准答案中的敏感词若出现在图片中，必须人工判断其是否为题干本来就要求展示的内容。

### 7.4 人工验收

生成 28 张图片的联系表和逐图审核报告。审核项包括：

- 结构与原题一致。
- 编号与空白位置正确。
- 无答案泄露。
- 无水印和推广信息。
- 中文无错字。
- 黑白风格一致。
- 缩放到实际答题页面宽度后仍清晰可读。

用户验收前不上传 OSS、不更新数据库。

## 8. OSS 上传设计

用户验收后，将 PNG 上传到：

```text
aisoftoj/questions/redrawn/<YYYY>/<MM>/<DD>/
```

SVG 源文件默认保留在本地；若用户希望在线保留源文件，可上传到独立 `source/` 子目录，但题库引用仍使用 PNG。

上传要求：

- `Content-Type` 正确设置为 `image/png`。
- 上传后逐张执行 HTTP 可访问性检查。
- 对内容计算 SHA-256，避免重复上传。
- 新 URL 写回 `manifest.json`，禁止依赖上传顺序进行映射。
- 任一图片上传失败时停止数据库迁移，保留已上传对象供重试，不产生半更新题库。

## 9. 数据库迁移与回滚

上传全部成功并完成用户验收后：

1. 导出所有受影响题目的 `id`、`intro`、`options`、`answer`、`analysis` 作为备份 SQL/JSON。
2. 根据 manifest 生成精确的旧 URL 到新 URL 映射。
3. 在事务中逐字段替换完整 URL，不做模糊域名替换。
4. 校验受影响题目数、替换次数和剩余旧域名引用数。
5. 提交事务后通过题目 API 和前端答题页抽查。

回滚使用备份数据按题目 ID 恢复原字段；OSS 新对象不立即删除，以免回滚后再次切换时丢失。

## 10. 完成标准

第一阶段完成标准：

- 28 张唯一图片均有 SVG 和 PNG 产物。
- 28 张均通过自动检查和人工检查。
- 联系表和审核报告可供用户逐张查看。
- 数据库仍保持原 URL，不发生线上或本地题库迁移。

第二阶段完成标准：

- 28 张 PNG 全部成功上传 OSS 并可访问。
- 33 次图片 URL 引用全部精确替换。
- 数据库中不再存在这 28 个旧图片 URL。
- 题目接口和答题页面显示正常。
- 备份与回滚文件完整可用。
